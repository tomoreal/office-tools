## 🎯 いつ分割するか（判断基準）

以下のいずれかに該当したら、分割を検討してください：

### **1. 保守が困難になった時**
- [ ] バグ修正に3日以上かかる
- [ ] どこに何があるか分からなくなった
- [ ] 修正の影響範囲が予測できない
- [ ] 同じようなコードを何度も書いている

### **2. 複数人で開発する時**
- [ ] 同じファイルを同時編集してgitコンフリクトが頻発
- [ ] 担当範囲を明確に分けたい
- [ ] コードレビューが困難

### **3. 大規模な機能追加がある時**
- [ ] 新しい出力形式追加（PDF、CSV など）
- [ ] Web API化
- [ ] 別システムとの連携
- [ ] 並列処理の本格導入

### **4. テストが必要になった時**
- [ ] ユーザーが増えて品質保証が必要
- [ ] 自動テスト（CI/CD）を導入したい
- [ ] リグレッションテストを書きたい

## 🔑 設計の3大原則（絶対遵守）

分割を実施する際は、以下の3つのルールを**絶対に守る**こと：

### **原則1: modelにロジックを入れない**

```python
# ❌ 禁止
class XBRLData:
    def calculate_total_assets(self):
        # 計算ロジック侵入
        pass

    def filter_by_period(self, period):
        # フィルタロジック侵入
        pass

# ✅ 正解（外部関数で実装）
def calculate_total_assets(data: XBRLData) -> float:
    # 計算ロジック
    pass

def filter_by_period(data: XBRLData, period: date) -> List[Fact]:
    # フィルタロジック
    pass
```

**理由**: modelは純粋なデータ構造のみ。ロジックを入れると肥大化し、テストが困難になる。

---

### **原則2: pipelineは「上から読める構造」を維持**

```python
# ❌ 禁止（スパゲッティ化）
def run_pipeline(...):
    # 500行の複雑なロジック...
    if error:
        retry...
    if option:
        ...

# ✅ 正解（一直線構造）
def run_pipeline(zip_paths, output_dir):
    """目次として機能"""
    files = load_phase(zip_paths, temp_dir)
    taxonomy = taxonomy_phase()
    data = parse_phase(files, taxonomy)
    return output_phase(data, output_dir)
```

**理由**: パイプラインは「処理の流れ」を示す目次。詳細は各フェーズ関数に委譲する。

---

### **原則3: sheetsでif分岐を書かない**

```python
# ❌ 禁止（分岐地獄）
def build_sheet(role_name, data):
    if role_name == "BalanceSheet":
        # BS専用処理...
    elif role_name == "ProfitOrLoss":
        # PL専用処理...
    elif is_ifrs:
        # IFRS処理...
    # 100行の分岐...

# ✅ 正解（Strategy Pattern）
SHEET_BUILDERS = {
    "BalanceSheet": build_balance_sheet,
    "ProfitOrLoss": build_profit_loss,
    "CashFlow": build_cashflow,
}

def build_sheet(role_name, data):
    builder = SHEET_BUILDERS.get(role_name, build_default)
    return builder(data)
```

**理由**: if文による分岐は保守性を大幅に下げる。テーブル駆動設計で解決する。

   ```bash

   rm convert_xbrl_to_excel.py
   # 新規ファイル大量作成...
   ```

   ```bash
   # ❌ 最も危険
   ```

   ```bash
   # ❌ 危険
   git add .
   git commit -m "全部分割した"
   ```

### **✅ 正しい進め方**

1. **1フェーズずつ**
   ```bash
   # ✅ 安全
   # Phase 1-1 完了 → 動作確認 → git commit
   # Phase 1-2 完了 → 動作確認 → git commit
   ```

2. **各フェーズ後の必須チェック**
   ```bash
   # ✅ 必須
   python3 convert_xbrl_to_excel.py sample.zip
   md5sum output.xlsx  # 分割前と比較
   ```

## 📖 用語集

| 用語 | 説明 |
|------|------|
| **Orchestrator** | 全体を統括する司令塔（processor.py） |
| **Pipeline** | 処理の流れを定義（pipeline.py） |
| **Strategy Pattern** | if文を使わず、辞書で処理を振り分ける設計パターン |
| **Pure Function** | 副作用を持たない関数（同じ入力なら同じ出力） |
| **Data Class** | ロジックを持たない純粋なデータ構造 |
| **LoggerAdapter** | ログにコンテキスト情報を追加する仕組み |
